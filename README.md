# github-deploy-pipeline

> Interactive GitHub release pipeline CLI — create staging pre-releases, promote to production, and sync branches, all from a single command.

[![npm version](https://img.shields.io/npm/v/github-deploy-pipeline)](https://www.npmjs.com/package/github-deploy-pipeline)
[![license](https://img.shields.io/npm/l/github-deploy-pipeline)](./LICENSE)
[![node](https://img.shields.io/node/v/github-deploy-pipeline)](https://nodejs.org)

---

## Overview

`github-deploy-pipeline` automates the GitHub release lifecycle through a guided interactive CLI:

- **Staging releases** — bump semver (RC, bug fix, minor, major, or custom), commit a version bump, create a GitHub pre-release with auto-generated notes, and optionally merge to your staging branch.
- **Production releases** — strip the `-rc.N` suffix from your latest RC, commit a clean version bump, create a final (non-prerelease) GitHub release, and optionally merge to your main branch.
- **Old release management** — list past RC releases and either promote one to production or re-publish it.

All release notes are auto-generated by the GitHub CLI, diffed against the previous release tag.

## Usage

```bash
npx gdeploy
```

Or add to your `package.json` scripts:

```json
{
  "scripts": {
    "deploy": "gdeploy"
  }
}
```

Then run:

```bash
npm run deploy
```

### Makefile integration

```makefile
deploy:
	npx gdeploy
```

## Interactive flow

```
🚀  GitHub Deploy Pipeline

? What would you like to do?
  ▶ Deploy new release
    Deploy old release

  → Deploy new release
    ? Deploy target
      ▶ Staging  (pre-release)
        Production  (final)

      → Staging
        ? Select bump type  (current latest: 1.2.3-rc.4)
          ▶ Release Candidate  →  1.2.3-rc.5
            Bug Fix            →  1.2.4-rc.1
            Minor              →  1.3.0-rc.1
            Major              →  2.0.0-rc.1
            Custom

      → Production
        Promotes 1.2.3-rc.4  →  1.2.3  (final, non-prerelease)

  → Deploy old release
    Lists all previous RC tags
    ? What to do with 1.2.1-rc.2?
      ▶ Promote to production  →  1.2.1  (final)
        Re-publish as same RC  →  1.2.1-rc.2
```

After each release you are offered to sync the relevant branch:

- **Staging release** → merge current branch → `staging` branch → push
- **Production release** → merge current branch → `main` branch → push

## Semver conventions

| Tag format | Meaning |
|------------|---------|
| `X.Y.Z-rc.N` | Staging pre-release |
| `X.Y.Z` | Final production release |

No `v` prefix is used. RC tags sort lower than final tags for the same `X.Y.Z`.

### Empty-repo behaviour

When no tags exist yet, the first staging release defaults to `0.1.0-rc.1` (or `1.0.0-rc.1` for a major bump). Release notes cover the full commit history.

## Configuration

Create `gdeploy.config.json` (or `.gdeployrc.json`) in your project root:

```json
{
  "versionFiles": [
    "package.json",
    "apps/backend/package.json",
    "apps/dashboard/package.json"
  ],
  "stagingBranch": "staging",
  "productionBranch": "main"
}
```

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `versionFiles` | `string[]` | `["package.json"]` | Relative paths to `package.json` files whose `version` field will be updated on each release. |
| `stagingBranch` | `string` | `"staging"` | Branch to sync after a staging release. |
| `productionBranch` | `string` | `"main"` | Branch to sync after a production release. |

Version files are committed automatically as `chore(release): <version>` before the GitHub release is created.

## Programmatic API

All internals are exported for use in custom scripts:

```ts
import {
  loadConfig,
  flowStaging,
  flowProduction,
  flowOldRelease,
  getTags,
  getLatestTag,
  parseSemVer,
  formatSemVer,
  bumpVersion,
} from 'github-deploy-pipeline';

const config = loadConfig(process.cwd());
const tags = getTags();
await flowStaging(tags, config, process.cwd());
```
